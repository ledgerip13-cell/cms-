## 一、问题清单（按严重程度排序）

### 🔴 严重（影响核心体验/数据完整性，建议优先修）

#### BUG-1：新增采集源无必填校验，可产生空白脏数据
- **现象**：admin「采集源管理」→「新增采集源」，不填任何字段直接点「保存」，成功创建一条 `name=''、apiUrl=''` 的空白记录。
- **根因**：`POST /api/sources`（`server/src/routes/sources.ts:19`）后端零校验；前端 `el-form` 未设 `rules`。
- **影响**：管理员误触即产生垃圾数据，污染采集源列表，可能被定时调度误采（空 apiUrl 会导致后续采集任务全部失败刷屏）。
- **复现**：打开新增弹窗→直接点保存。
- **测试中已产生1条脏数据并手动清理（Source id=8）**。
- **修复建议**：后端加 `if (!b.name?.trim() || !b.apiUrl?.trim()) return 400`；前端 `el-form-item` 加 `:rules="[{required:true,message:'必填'}]"` + `prop`。

#### BUG-2：影片详情不存在时前端永久卡在骨架屏，无错误提示
- **现象**：访问 `/#/play/{不存在的id}`，页面永久显示灰色骨架屏动画，不会降级为"影片不存在"提示，用户会误以为卡死/网络问题。
- **根因**：后端 `GET /api/vods/:id` 对不存在的记录返回 `200 {error:"not found"}`（而非 404），前端 `Play.vue` 用 `v-if="vod.id"` 判断加载完成，`error` 字段没有被消费，`vod.id` 永远是 `undefined`，卡在 `v-else` 骨架屏分支出不来。
- **影响**：死链、已下架、被误删的影片ID，用户看到的是"无限加载中"而非明确提示，体验差且不可自证。
- **修复建议**：后端对 not found 返回 `reply.code(404)`；前端 `loadVod` 捕获 error/404，展示"该影片不存在或已下架"提示 + 返回首页按钮。

#### BUG-3：每日更新板块被同名剧集刷屏，多样性缺失
- **现象**：首页「每日更新」板块14个坑位里，同一天出现6条"闪电侠"（第一季~第九季等不同季度独立记录）。
- **根因**：`GET /api/weekly`（`server/src/routes/vods.ts:82`）取最近7天 `updatedAt` 降序600条按星期分组，**未做同名影片去重/打散**，当某天批量采集同一IP的多季内容时会扎堆占满当天名额，挤掉其他内容。
- **影响**：板块观感重复、单调，实际"多样性推荐"的产品意图落空。
- **修复建议**：分组时按 `normalizeName(name)` 去重（同名只保留最新一条），或改成"同名最多出现1次"的限流逻辑。

### 🟡 中等（明显但有退路，或范围有限）

#### BUG-4：移动端播放页可能残留侧边栏遮挡全部内容
- **现象**：在移动端宽度（390px）下，若用户此前在首页点开过汉堡菜单（`drawer=true`），随后点击海报卡片跳转到播放页，侧边栏抽屉状态未重置，整个播放器/线路/选集被完全遮挡，只露出右侧"相关推荐"。
- **根因**：`App.vue` 里 `drawer.value=false` 只在 `goHome()`/`pick()`（点导航项）时重置，海报卡片走的 `goPlay(id)` 跳转路径完全没有关闭抽屉的逻辑。
- **退路**：点击背景遮罩层（`.sb-backdrop`）可以关闭，不是彻底卡死，但首次遭遇会让用户以为白屏/加载失败。
- **修复建议**：在 `router.afterEach` 或 `watch(route)` 里统一强制 `drawer.value=false`，不要依赖各入口各自处理。

#### BUG-5：年份筛选器硬编码只显示16个年份，老片筛不到
- **现象**：浏览页/首页筛选栏「年份」只列出 2013~2030（含异常的2030、2027未来年份），2012年及更早的片子（现存约500+部，最早到2003年附近）完全没有对应筛选按钮。
- **根因**：`GET /api/years`（`server/src/routes/vods.ts:35`）`.slice(0, 16)` 硬编码只取降序前16个年份。
- **影响**：老片无法通过年份精确筛选，只能翻页硬找。
- **修复建议**：年份列表改成滚动横排（不限16个）或做年代分组（2020s/2010s/2000s及以前）代替逐年平铺。

#### BUG-6：admin 采集源管理表格列被裁切不可见
- **现象**：「启用」「定时采集」「累计入库」「最近采集」四列在1280px宽度下完全看不到，也没有横向滚动条可以拉出来查看。
- **根因**：`el-table` 列总宽度（约1410px）超过父容器 `.card`（约1012px），横向滚动未生效（`wrapperOverflowX` 计算结果为 `hidden`），列被裁切丢失而非可滚动查看。
- **影响**：管理员看不到源的启用状态、定时采集配置、累计入库量、最近采集时间这些关键运维信息，只能进编辑弹窗才能看到。
- **修复建议**：给 `.el-table` 包一层 `overflow-x:auto` 容器；或收窄部分列宽（如"最近采集"从160px降到120px，采集API从240px降到180px），使总宽度回落到容器内。

### 🟢 轻微 / 内容配置类（非代码缺陷，运营侧待办）

#### ISSUE-7：站点描述/页脚残留未替换的模板占位符
- 「网站描述」字段内容为 `欢迎来到【网站名称】——为您打造的终极影视...`，`【网站名称】` 是模板变量没有被替换成实际站名，已直接发布到线上前端和SEO描述里。
- 「页脚文字」用了 📅（日历emoji）代替版权符号 `©`，且紧跟着又写了一遍 `©`，变成 `📅 Copyright © 2025-2026`，图标语义不当+文案重复。
- **建议**：admin「站点设置」页手动重新编辑这两处文案。

#### ISSUE-8：豆瓣元数据匹配率仅 0%（13/11745），评分数据几乎全部缺失
- 影响：首页 Hero、热搜/国漫/动漫/短剧榜的"评分优先"排序逻辑因为几乎没有评分数据，实际全部退化成"最近更新"兜底排序，产品设计的"口碑榜"名不副实。
- **建议**：admin「元数据」页点「匹配未处理（11693部）」批量跑一次（当前配置：限速2500ms/条、每批50部，11693部预计需要约8小时不间断跑批，建议调大 `batchLimit` 或分多轮夜间执行）。

#### ISSUE-9：如意资源网有10项源分类未映射
- 影响：这10个源分类的片子会走分类器兜底猜测，可能被错误归类。
- **建议**：admin「分类映射」页逐个手动映射到统一分类。

#### ISSUE-10：数据中存在异常未来年份（2030、2027等）
- 库中有 `year=2030`、`year=2027` 的记录（各1~2条），明显是源站脏数据或未上映片的占位年份，未做合理性校验直接入库。
- **建议**：采集时加年份合理性校验（`year <= 当前年份+1`），超出范围的标记待复核而非直接采信。

#### ISSUE-11：admin 后台未做移动端适配（设计如此，非缺陷）
- 移动端（390px）访问 admin 时侧边栏固定展开挤占主内容，表格文字被压缩到竖排不可用。
- **判定**：后台管理系统通常仅面向桌面端使用，这是合理的产品定位，**不建议**列入bug修复优先级，仅作记录。

## 二、验证通过、无异常的项目

- web端首页/浏览页/播放页在桌面端和移动端（390px）均**无JS运行时报错**（`window.onerror` / `unhandledrejection` / `console.error` 全程监听为空）。
- 搜索空结果场景正确展示"没有找到相关影片"，无报错无卡死。
- 移动端首页布局、卡片网格自适应、汉堡菜单抽屉交互（进入/退出）本身工作正常。
- admin Dashboard 统计数字自洽（11745影片总数 vs 11744在线，差值=1条下架片，合理）。
- 采集任务并发保护逻辑正确（同源任务运行中再次提交会被正常拒绝，非bug）。
- 按片名两步式采集（预览→勾选→精确入库）功能链路完整验证通过，未见异常。

## 三、升级建议（按优先级）

### P0（本周内处理，涉及数据完整性/核心体验）
1. 修复 BUG-1（新增采集源必填校验）—— 防止继续产生脏数据
2. 修复 BUG-2（影片不存在时的友好提示）—— 用户体验底线
3. 修复 BUG-3（每日更新去重打散）—— 影响首页核心展示质量

### P1（近期处理，明显缺陷但有临时退路）
4. 修复 BUG-4（移动端侧边栏状态重置）
5. 修复 BUG-5（年份筛选器移除16条硬限制）
6. 修复 BUG-6（admin表格横向滚动/列宽收窄）

### P2（运营/内容侧待办，非代码修复）
7. 重新编辑站点描述/页脚文案（ISSUE-7）
8. 批量执行豆瓣元数据匹配，把评分覆盖率从0.1%提升到可用水平（ISSUE-8）—— 建议先小批量试跑确认限速稳定，再排队夜间批量执行
9. 补全如意资源网剩余10项分类映射（ISSUE-9）
10. 采集时加年份合理性校验（ISSUE-10）

### 长期优化方向（非本次测试发现的bug，是观察到的产品机会点）
- 当前"热门榜"排序本质是"评分人数优先"，但评分覆盖率极低时该逻辑基本不生效，建议在评分数据补齐前，临时增加"播放线路数 × 探活健康分"作为热度代理指标，比纯"最近更新"更有说服力。
- admin 缺少批量操作（批量下架/批量重新采集），11745部体量下逐条操作效率较低，可考虑列表多选+批量操作栏。
- 建议给关键表单（新增/编辑采集源、站点设置）补充前端字段级校验和后端参数校验双保险，这次空白源问题只是冰山一角，其他表单未逐一穷举测试，建议做一次专项表单校验审计。

## 四、测试方法说明（供复核参考）

- JS报错捕获：`window.addEventListener('error'/'unhandledrejection')` + hook `console.error`，全程未捕获到任何异常，说明发现的问题均为**逻辑/数据层面缺陷，非代码崩溃类问题**。
- 响应式测试：浏览器视口 1280×800（桌面）与 390×844（iPhone常见尺寸）两档。
- 数据交叉验证：所有涉及数据现象的问题均用 `docker exec vcms-postgres psql` 直查数据库佐证，非仅凭UI猜测。
- 边界场景覆盖：不存在的影片ID、空搜索关键词、空表单提交、跨页面状态残留。
